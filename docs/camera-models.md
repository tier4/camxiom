# Camera models

This page is the specification of camxiom's data model: coordinate
conventions, the `CameraModel` aggregate, and the exact parameter layout of
every supported projection and distortion model.

## Conventions

These conventions apply to the entire API.

### Camera coordinate frame

Right-handed, **+Z forward** (the optical axis, pointing out of the camera
into the scene), **+X right**, **+Y down**. Rays with `z <= 0` are
`BEHIND_CAMERA` for pinhole; wide-angle models (fisheye / omnidirectional /
double sphere / EUCM) accept rays up to their `theta_max`.

Projection is central: ray *direction* is what matters. Input rays do not
need to be unit-normalized (the magnitude cancels in every projection
formula), and the `Ray3` overloads ignore `ray.origin`.

### Pixel coordinates

`u` along +X (columns, increasing rightward), `v` along +Y (rows, increasing
downward). **Pixel-center origin**: integer coordinate `(0, 0)` is the
*center* of the top-left pixel, so the geometric center of a `W × H` image is
`((W-1)/2, (H-1)/2)`. Rectified output models generated by camxiom use this
convention for their principal point.

### Intrinsics

`fx, fy, cx, cy, skew` are in pixels. With distorted normalized coordinates
`(x_d, y_d)`:

```
u = fx * x_d + skew * y_d + cx
v = fy * y_d              + cy
```

(skew multiplies `y_d` in the `u` equation only; most cameras have skew = 0.)

### Angles

Every angle in the API is in **radians** unless the name says otherwise (only
`*_fov_deg` fields are degrees). `theta` always denotes the angle between a
ray and the optical axis.

## The `CameraModel` aggregate

```cpp
struct CameraModel {                 // = CameraModelT<float>; CameraModel64 = CameraModelT<double>
  IntrinsicsModel intrinsics;        // fx, fy, cx, cy, skew            (pixels)
  ProjectionModel projection;        // type, theta_max, xi, alpha, beta
  DistortionModel distortion;        // type, space, coeffs[14], tilt matrices, count, flags
};
```

All model types are plain aggregates (no constructors): brace-initializable,
stack-allocatable (~200 bytes), no virtual functions, no heap. A camera model
is the *composition* of three orthogonal parts:

- **`IntrinsicsModel`** — the affine pixel mapping (`fx`, `fy`, `cx`, `cy`, `skew`).
- **`ProjectionModel`** — the projection family and its *projection-level*
  parameters: `type`, `theta_max` (maximum accepted ray-to-axis angle,
  default π/2), `xi` (omnidirectional / double-sphere), `alpha`
  (double-sphere / EUCM), `beta` (EUCM).
- **`DistortionModel`** — the distortion model applied in between: `type`,
  `space` (`PLANE` or `ANGLE`), `coeffs[14]` (OpenCV-compatible layout,
  `count` says how many leading entries are meaningful), precomputed tilt
  matrices for TILTED14, and derived flags (`is_rational`, `has_thin_prism`,
  `has_tilt`).

### Float and double variants

Every type is a single `*T<T>` template instantiated for both precisions:

| float32 (primary, real-time) | float64 (calibration / optimization) |
|---|---|
| `CameraModel`, `IntrinsicsModel`, `ProjectionModel`, `DistortionModel` | `CameraModel64`, `IntrinsicsModel64`, ... |
| `Pixel2`, `Ray3`, `PixelResult`, `RayResult` | `Pixel2d`, `Ray3d`, `PixelResult64`, `RayResult64` |
| `SolverOptions` (10 iters, 1e-6) | `SolverOptions64` (15 iters, 1e-12 — deliberately a distinct struct with tighter defaults) |

Member names are identical across precisions. Conversion helpers:
`toCameraModel64()`, `fromCameraModel64()`, `toSolverOptions64()`.

## Projection models

`ProjectionModelType`: `PINHOLE`, `FISHEYE_THETA`, `OMNIDIRECTIONAL`,
`DOUBLE_SPHERE`, `EUCM` (+ `UNKNOWN`).

| Model | Projection-level params | Forward projection (before distortion & intrinsics) | Reference |
|---|---|---|---|
| `PINHOLE` | — | `(x, y) = (X/Z, Y/Z)` | — |
| `FISHEYE_THETA` | `theta_max` | `θ = atan2(√(X²+Y²), Z)`; radius `r = f·θ_d(θ)` per the angle-distortion model | Kannala & Brandt, PAMI 2006 |
| `OMNIDIRECTIONAL` (Mei) | `xi` | `(x, y) = (X, Y) / (Z + ξ·‖P‖)` | Mei & Rives, ICRA 2007 |
| `DOUBLE_SPHERE` | `xi`, `alpha` | `d₁=‖P‖`, `d₂=√(X²+Y²+(ξd₁+Z)²)`, `(x, y) = (X, Y) / (α·d₂ + (1−α)(ξd₁+Z))` | Usenko et al., 3DV 2018 |
| `EUCM` | `alpha`, `beta` | `d=√(β(X²+Y²)+Z²)`, `(x, y) = (X, Y) / (α·d + (1−α)Z)` | Khomutenko et al., RA-L 2015 |

Domain limits are enforced, not clamped: the Mei model rejects directions
past its injectivity limit (`‖P‖ + ξZ > 0`), double sphere enforces the
bijectivity condition of Usenko 2018 (eq. 43–45), and EUCM enforces its FOV
validity condition. Out-of-domain rays return a non-`OK` status rather than
folding onto a wrong pixel.

Typical parameter ranges: Mei `ξ ≥ 0` (1.0 = parabolic mirror); double
sphere `ξ ∈ (−1, 1)` with real cameras typically in `[−0.4, −0.05]`,
`α ∈ (0, 1)`; EUCM `α ∈ (0, 1)`, `β > 0` (1.0 = unified camera model).

### `theta_max`

`projection.theta_max` bounds the accepted ray-to-axis angle (default π/2; a
full-hemisphere-plus fisheye uses up to π). For fisheye models it also bounds
the monotonic range used by the inverse theta solver, so it must stay
consistent with the distortion coefficients:

```cpp
void updateThetaMax(CameraModel &model);
```

must be called after distortion coefficients are modified by hand or by an
optimizer (no-op for non-fisheye models). Model factories
(`makeCameraModel`, `getDefaultSeed`, `calibrate`, `convertCameraModel`)
already leave `theta_max` consistent.

This consistency is enforced, not advisory: for the polynomial families
(KB4 / KB8 / OpenCV-fisheye4) `validateCameraModel` certifies that
`theta_max` sits inside the polynomial's positive monotone range, and even
the per-point query APIs reject a cap whose endpoint slope is negative
(the common hand-set-cap-past-the-fold case) as `INVALID_MODEL`. Likewise,
the coefficient-free types (`NONE` and the ideal `equidistant` /
`equisolid` / `stereographic` / `orthographic` mappings) must carry an
all-zero `coeffs` array — stale non-zero entries are rejected instead of
being silently ignored by some code paths and applied by others.

## Distortion models

`DistortionSpace` divides distortion models into two families, and
`validateCameraModel` rejects mismatched combinations (e.g. a pinhole
projection with an angle-space distortion) instead of producing garbage:

- **`PLANE`** — distortion applied to normalized image-plane coordinates
  (pinhole family; also the optional add-on distortion of Mei / DS / EUCM).
- **`ANGLE`** — distortion applied to the incidence angle θ (fisheye family).

### PLANE space

Coefficient array layout (OpenCV-compatible, full TILTED14 ordering):

```
[k1, k2, p1, p2, k3, k4, k5, k6, s1, s2, s3, s4, tau_x, tau_y]
```

| `DistortionModelType` | `count` | Meaning |
|---|---|---|
| `NONE` | 0 | distortion-free |
| `RADTAN4` | 4 | radial `k1,k2` + tangential `p1,p2` |
| `RADTAN5` | 5 | + `k3` (OpenCV `plumb_bob` / Brown–Conrady) |
| `RATIONAL8` | 8 | rational model; `k4..k6` form the denominator |
| `THIN_PRISM12` | 12 | + thin-prism `s1..s4` |
| `TILTED14` | 14 | + Scheimpflug tilt `tau_x, tau_y` (radians). The 3×3 tilt matrix and its inverse are precomputed at model-build time — no per-point trigonometry. |

### ANGLE space

Distortion acts on the incidence angle: the projected radius is
`r = f · θ_d(θ)`.

| `DistortionModelType` | Coeffs | `θ_d(θ)` |
|---|---|---|
| `OPENCV_FISHEYE4` | `[k1..k4]` | `θ·(1 + k1θ² + k2θ⁴ + k3θ⁶ + k4θ⁸)` (OpenCV fisheye) |
| `KB4` | `[k1..k4]` | `θ + k1θ³ + k2θ⁵ + k3θ⁷ + k4θ⁹` (Kannala–Brandt) |
| `KB8` | `[k1..k8]` | `θ + k1θ³ + ... + k8θ¹⁷` |
| `EQUIDISTANT` | — | `θ` (ideal, coefficient-free) |
| `EQUISOLID` | — | `2·sin(θ/2)` |
| `STEREOGRAPHIC` | — | `2·tan(θ/2)` |
| `ORTHOGRAPHIC` | — | `sin(θ)` (rear hemisphere rejected — non-injective past π/2) |

### Projection-level parameters are *not* distortion coefficients

`xi`, `alpha`, `beta` live in `ProjectionModel`, never in `coeffs`.
Interchange formats that pack them into a `D` vector (e.g. a
`sensor_msgs/CameraInfo` for a double-sphere camera) are unpacked by the
[CameraInfo interchange layer](api/camera-info.md): DS reads
`d = [xi, alpha, plane-coeffs...]`, EUCM `d = [alpha, beta, ...]`, Mei
`d = [xi, ...]`.

## Validation

```cpp
[[nodiscard]] StatusCode validateCameraModel(const CameraModel &model);
[[nodiscard]] StatusCode validateCameraModel64(const CameraModel64 &model);
```

Checks internal consistency (finite parameters, valid type/space
combinations, parameter ranges such as DS `ξ ∈ (−1, 1)`, `theta_max`
consistency). Both precisions are single-sourced from one validator template,
so a model accepted in float is accepted after widening to double — see
[Numerical strategy](design/numerical-strategy.md) for why this is enforced
structurally.

The generic projection functions validate the model on every call. For hot
loops, validate once via `ValidatedCameraModel::tryMake()` — see
[Projection](api/projection.md).
